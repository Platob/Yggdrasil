# Workspace instructions — Yggdrasil (`ygg[data,databricks]`)

This workspace uses [Yggdrasil](https://github.com/Platob/Yggdrasil)
(PyPI: `ygg`, import: `yggdrasil`) as the canonical surface for moving
data between Python types, Arrow, Polars, pandas, Spark, and Databricks.
Generated code should integrate with it instead of reinventing parallel
helpers.

## Install line in notebooks

```python
%pip install "ygg[data,databricks]"
dbutils.library.restartPython()
```

`[data]` pulls pandas + numpy + sqlglot; `[databricks]` pulls
`databricks-sdk`. Spark is already on the cluster — do **not** add
`[bigdata]` on a Databricks runtime.

## Always reach for `yggdrasil.data` before raw engine APIs

`yggdrasil.data` is the canonical surface for describing and moving
frames. Prefer these primitives, in this order, before dropping down to
`pyarrow` / `polars` / `pandas` / `pyspark`:

- `yggdrasil.data.cast.convert(value, target)` for any value → type
  coercion. It dispatches a single registry (exact match → identity →
  `Any` wildcard → MRO → one-hop composition). Don't hand-roll
  `int(x)`/`str(x)`/`pd.to_datetime(x)` chains when `convert` already
  handles them.
- `yggdrasil.data.DataField` / `Schema` for column descriptions —
  carries name, nullability, metadata, nested structure, and per-engine
  dtype intent in one place.
- `yggdrasil.data.DataType` / `DataTypeId` for type hints — produces
  pyarrow / polars / pandas / Spark dtypes from one source.
- `yggdrasil.data.DataTable` / `StatementResult` for "execute, then
  move rows somewhere" workflows.

## Always go through `DatabricksClient` for platform calls

```python
from yggdrasil.databricks import DatabricksClient

dbc = DatabricksClient()  # picks up DATABRICKS_HOST/TOKEN from env
```

`DatabricksClient` is a singleton-by-config: same `(host, auth)` →
same instance, picklable across Spark workers / job tasks. Reach for
its services rather than `databricks.sdk.WorkspaceClient` directly:

| You want… | Use |
| --- | --- |
| Run SQL | `dbc.sql.execute(q)` → `StatementResult` |
| Catalog / schema / table metadata | `dbc.catalogs`, `dbc.schemas`, `dbc.tables` |
| Volumes / DBFS / Workspace files | `dbc.volumes`, `dbc.dbfs_path(...)`, `dbc.workspace_path(...)` |
| Compute / warehouses / clusters | `dbc.compute`, `dbc.warehouses`, `dbc.clusters` |
| Jobs | `dbc.jobs` |
| Secrets | `dbc.secrets` |
| IAM | `dbc.iam` |
| Genie | `dbc.genie` |

## House rules for generated code

- **No row-by-row Python loops over data.** Use `pyarrow.compute`
  kernels, Polars expressions, or numpy ufuncs before reaching for a
  `for` loop or `.to_pylist()` / `.tolist()` / `.to_list()`. Heavy
  per-row materialisation is the single biggest perf trap.
- **Cross engines via zero-copy bridges:** `Array.to_pandas()`,
  `pl.from_arrow(arr)`, `series.to_arrow()`, `pl.from_pandas(df)`. Do
  **not** loop or `to_pylist` to bounce between engines.
- **Pass `CastOptions(target_field=...)` to lock a schema** rather than
  chaining `.cast()` calls per column.
- **Use `yggdrasil.pickle.json` for JSON**, not stdlib `json`. It wraps
  `orjson` with type coverage for datetime / UUID / Path / Enum /
  dataclass / Decimal.
- **Use `yggdrasil.data.enums`** for fixed-vocabulary tokens
  (`MimeTypes`, `Scheme`, `ByteUnit`, currency, timezone, geozone)
  instead of regex / dict / `int(x) * 1024` at call sites.
- **Optional deps live behind `lib.py` guards.** Use
  `from yggdrasil.polars.lib import polars` (or
  `yggdrasil.pandas.lib` / `yggdrasil.spark.lib` /
  `yggdrasil.databricks.lib`). Plain `import polars` works on a runtime
  cluster but breaks base installs.

## Logging shape

The library follows: `<Verb> <ResourceNoun> %r (key=value, …)` with
present-progressive at the start (`"Creating"`, `"Deleting"`) and past
tense on completion (`"Created"`, `"Deleted"`). Generated logging
should match that scannable shape — and use `%r` (lazy) over f-strings,
so disabled levels skip the `repr()`.
