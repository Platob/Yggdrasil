# Workspace instructions ÔÇö Yggdrasil (`ygg[data,databricks]`)

This workspace uses [Yggdrasil](https://github.com/Platob/Yggdrasil)
(PyPI: `ygg`, import: `yggdrasil`) as the **mandatory** library for
moving data between Python types, Arrow, Polars, pandas, Spark, and
Databricks. **Every generated notebook, script, and job task on this
workspace must use `ygg[data,databricks]`** ÔÇö it is not optional, and
not one of several alternatives.

## Hard rules (non-negotiable)

1. **The `ygg` package is required.** Generated code that depends on
   data movement, schema handling, HTTP, Databricks integration,
   casting, or any of the surfaces below must reach through
   `yggdrasil`. **Do not** ship code that imports `requests`,
   `databricks.sdk.WorkspaceClient` directly, or hand-rolls
   `pd.to_datetime(...)` / `int(x)` casts when the canonical path
   exists.
2. **Install line is fixed.** Every notebook starts with the install
   block below ÔÇö no exceptions:
   ```python
   %pip install "ygg[data,databricks]"
   dbutils.library.restartPython()
   ```
3. **Imports route through `yggdrasil.*`.** Bare `import requests`,
   `import databricks.sdk`, `from databricks.sdk.service.x import ÔÇŽ`
   in feature code is rejected. Go through `dbc.<service>` and the
   `yggdrasil.io.http_` / `yggdrasil.data` modules.
4. **No parallel helpers.** Don't create a new utility module when
   an existing `yggdrasil` function does most of the job ÔÇö extend
   the existing function.

## Autonomy expectations

This assistant is configured to act autonomously on data-engineering
and analytics tasks. **Default to acting, not asking.** The expected
loop on any non-trivial task:

1. **Plan.** State the steps in 3-7 bullets before writing code.
   Identify which yggdrasil skills (`ygg-*`) the task touches.
2. **Think longer on edge cases.** Schema drift, idempotency, timezone
   intent, decimal precision, FK target existence, pagination cursors,
   429 retry exhaustion, partial-batch failures. Don't ship without
   considering them.
3. **Benchmark when touching a hot path.** Cast registry, IO buffers,
   HTTP send, table insert, schema reconciliation, transform loops in
   curated views. See `ygg-benchmarks`. Quote before/after numbers in
   the commit body.
4. **Smoke-test before declaring done.** Unit tests for logic, a live
   one-batch run for ingestion pipelines, schema reconciliation on a
   throwaway table for DDL changes. **Don't say "done" without proof
   the change actually works.**
5. **Only stop and ask** when something is genuinely ambiguous
   (idempotency strategy, overwrite vs append, which secret scope).
   Pick a defensible default and document it in a one-line comment
   for everything else.

## Install line in notebooks

```python
%pip install "ygg[data,databricks]"
dbutils.library.restartPython()
```

`[data]` pulls pandas + numpy + sqlglot; `[databricks]` pulls
`databricks-sdk`. Spark is already on the cluster ÔÇö do **not** add
`[bigdata]` on a Databricks runtime.

## Always reach for `yggdrasil.data` before raw engine APIs

`yggdrasil.data` is the canonical surface for describing and moving
frames. Prefer these primitives, in this order, before dropping down to
`pyarrow` / `polars` / `pandas` / `pyspark`:

- `yggdrasil.data.cast.convert(value, target)` for any value Ôćĺ type
  coercion. It dispatches a single registry (exact match Ôćĺ identity Ôćĺ
  `Any` wildcard Ôćĺ MRO Ôćĺ one-hop composition). Don't hand-roll
  `int(x)`/`str(x)`/`pd.to_datetime(x)` chains when `convert` already
  handles them.
- `yggdrasil.data.Field` / `Schema` for column descriptions ÔÇö
  carries name, nullability, metadata, nested structure, and per-engine
  dtype intent in one place.
- `yggdrasil.data.DataType` / `DataTypeId` for type hints ÔÇö produces
  pyarrow / polars / pandas / Spark dtypes from one source.
- `yggdrasil.data.DataTable` / `StatementResult` for "execute, then
  move rows somewhere" workflows.

## Always go through `DatabricksClient` for platform calls

```python
from yggdrasil.databricks import DatabricksClient

dbc = DatabricksClient()  # picks up DATABRICKS_HOST/TOKEN from env
```

`DatabricksClient` is a singleton-by-config: same `(host, auth)` Ôćĺ
same instance, picklable across Spark workers / job tasks. Reach for
its services rather than `databricks.sdk.WorkspaceClient` directly:

| You wantÔÇŽ | Use |
| --- | --- |
| Run SQL | `dbc.sql.execute(q)` Ôćĺ `StatementResult` |
| Catalog / schema / table metadata | `dbc.catalogs`, `dbc.schemas`, `dbc.tables` |
| Volumes / DBFS / Workspace files | `dbc.volumes`, `dbc.dbfs_path(...)`, `dbc.workspace_path(...)` |
| Compute / warehouses / clusters | `dbc.compute`, `dbc.warehouses`, `dbc.clusters` |
| Jobs | `dbc.jobs` |
| Secrets | `dbc.secrets` |
| IAM | `dbc.iam` |
| Genie | `dbc.genie` |

## House rules for generated code

- **No row-by-row Python loops over data.** Use `pyarrow.compute`
  kernels, Polars expressions, or numpy ufuncs before reaching for a
  `for` loop or `.to_pylist()` / `.tolist()` / `.to_list()`. Heavy
  per-row materialisation is the single biggest perf trap.
- **Cross engines via zero-copy bridges:** `Array.to_pandas()`,
  `pl.from_arrow(arr)`, `series.to_arrow()`, `pl.from_pandas(df)`. Do
  **not** loop or `to_pylist` to bounce between engines.
- **Cross pipeline stages in Arrow / Spark batches, not rows.**
  When two stages share a schema (paginated HTTP Ôćĺ cast Ôćĺ write,
  Job task Ôćĺ Job task, SQL result Ôćĺ next transform, Kafka /
  Kinesis batch Ôćĺ frame Ôćĺ insert), the unit on the wire is a
  `pa.Table` / `pa.RecordBatch` / Spark `DataFrame` ÔÇö never
  `Iterable[dict]`, never `.collect()` Ôćĺ list of Row, never
  `.iter_rows(named=True)` between two pipeline stages that both
  know the schema. Use Arrow IPC (`pa.ipc.RecordBatchStreamWriter`)
  for cross-process / cross-language transport. Per-row dicts are
  for **genuine row endpoints** only (ndjson / xlsx writers, Kafka
  producers, JSON HTTP responses where one element = one row).
  Full rule: AGENTS.md ┬ž6b.
- **Pass `CastOptions(target_field=...)` to lock a schema** rather than
  chaining `.cast()` calls per column.
- **Use `yggdrasil.pickle.json` for JSON**, not stdlib `json`. It wraps
  `orjson` with type coverage for datetime / UUID / Path / Enum /
  dataclass / Decimal.
- **Use `yggdrasil.data.enums`** for fixed-vocabulary tokens
  (`MimeTypes`, `Scheme`, `ByteUnit`, currency, timezone, geozone)
  instead of regex / dict / `int(x) * 1024` at call sites.
- **Geographic data is always renderable.** Every curated table
  carrying a location reference (country, region, exchange,
  bidding zone, station, route, route point) ships `lat: float64
  [-90, 90]` + `lon: float64 [-180, 180]` (WGS84) so map plugins
  can plot rows directly. Zone / region polygons add
  `boundary_geojson: string` (a GeoJSON `Feature` /
  `FeatureCollection`). Tracks emit `points: list<struct<lat, lon,
  observation_utc>>`. Resolve at curate time via
  `GeoZoneCatalog` ÔÇö every `GeoZone` already carries `lat` /
  `lon`. See `ygg-curated-views` ┬ž "Geographic data Ôćĺ always
  carry lat/lon".
- **Optional deps live behind `lib.py` guards.** Use
  `from yggdrasil.polars.lib import polars` (or
  `yggdrasil.pandas.lib` / `yggdrasil.spark.lib` /
  `yggdrasil.databricks.lib`). Plain `import polars` works on a runtime
  cluster but breaks base installs.

## Logging shape

The library follows: `<Verb> <ResourceNoun> %r (key=value, ÔÇŽ)` with
present-progressive at the start (`"Creating"`, `"Deleting"`) and past
tense on completion (`"Created"`, `"Deleted"`). Generated logging
should match that scannable shape ÔÇö and use `%r` (lazy) over f-strings,
so disabled levels skip the `repr()`.
